# review

对前端代码进行全面的 Code Review，检查代码质量、性能、安全性和最佳实践。

## 执行流程

### 1. 收集变更信息（并行执行）

运行以下命令了解代码变更：
- `git status` - 查看所有修改的文件
- `git diff` - 查看具体代码变更
- `git diff --cached` - 查看已暂存的变更

### 2. 代码审查维度

按以下维度对代码进行审查，发现问题时立即指出并提供修复建议：

#### A. 代码质量 (Code Quality)

**命名规范：**
- [ ] 组件文件使用 PascalCase（如 `Button.jsx`）
- [ ] 工具函数使用 camelCase（如 `formatDate.js`）
- [ ] 常量使用 UPPER_SNAKE_CASE（如 `API_BASE_URL`）
- [ ] CSS 类名使用 kebab-case（如 `btn-primary`）
- [ ] 变量和函数名语义清晰，避免缩写

**代码结构：**
- [ ] 每个文件只导出一个主要组件
- [ ] 文件长度不超过 200 行（超过需拆分）
- [ ] 导入语句按类型分组（React → 第三方库 → 本地组件 → 工具函数）
- [ ] 删除未使用的导入和变量
- [ ] 避免使用 `eslint-disable` 注释

**组件结构顺序：**
- [ ] 1. 导入依赖
- [ ] 2. 组件定义
- [ ] 3. Hooks（useState → useEffect → 自定义 hooks）
- [ ] 4. 事件处理函数
- [ ] 5. 渲染逻辑
- [ ] 6. PropTypes 定义
- [ ] 7. 默认导出

#### B. React 组件规范 (React Best Practices)

**组件设计：**
- [ ] 单一职责原则：每个组件只做一件事
- [ ] Props 驱动：通过 props 控制组件行为和样式
- [ ] 避免副作用：不直接修改外部状态
- [ ] 可复用性：通用组件放在 `components/common/`
- [ ] PropTypes 或 TypeScript 类型定义完整

**Hooks 使用：**
- [ ] 不在循环、条件或嵌套函数中调用 Hooks
- [ ] useEffect 依赖数组正确且完整
- [ ] 避免在 render 中创建新函数（使用 useCallback）
- [ ] 避免在 render 中创建新对象（使用 useMemo）
- [ ] 自定义 Hooks 命名以 `use` 开头

**状态管理：**
- [ ] 局部状态使用 useState
- [ ] 共享状态提升到最近的公共父组件
- [ ] 避免过度状态：能通过 props 计算的不存为 state
- [ ] 避免重复状态：能通过现有 state 派生的不重复存储

#### C. 性能优化 (Performance)

**必须遵守：**
- [ ] 所有页面组件使用 `React.lazy()` 懒加载
- [ ] 图片使用 `loading="lazy"` 属性
- [ ] 列表渲染必须添加唯一 `key` 属性（不使用 index）
- [ ] 避免在 render 中创建新函数或对象
- [ ] 大型列表考虑虚拟滚动（react-window）

**推荐实践：**
- [ ] 使用 `useMemo` 缓存计算结果
- [ ] 使用 `useCallback` 缓存回调函数
- [ ] 图片使用 WebP 格式，提供降级方案
- [ ] 使用 Code Splitting 按需加载第三方库
- [ ] 避免不必要的重渲染（使用 React.memo）

#### D. 安全性 (Security)

**常见漏洞检查：**
- [ ] 防止 XSS：用户输入必须转义，避免 `dangerouslySetInnerHTML`
- [ ] 防止 SQL 注入：使用参数化查询
- [ ] 防止命令注入：避免直接执行用户输入
- [ ] 敏感信息：API key、密码不得硬编码或提交
- [ ] HTTPS：所有外部请求使用 HTTPS
- [ ] 依赖安全：检查是否有已知漏洞的依赖

**数据验证：**
- [ ] 用户输入必须验证（类型、长度、格式）
- [ ] API 响应必须验证（防止恶意数据）
- [ ] 文件上传必须验证（类型、大小）

#### E. 可访问性 (Accessibility)

**基础要求：**
- [ ] 图片必须有 `alt` 属性
- [ ] 按钮和链接有清晰的文本或 `aria-label`
- [ ] 表单输入有对应的 `<label>` 或 `aria-label`
- [ ] 颜色对比度符合 WCAG AA 标准（至少 4.5:1）
- [ ] 键盘可访问：所有交互元素可通过 Tab 键访问

**进阶要求：**
- [ ] 使用语义化 HTML（header, nav, main, footer）
- [ ] 动态内容变化使用 `aria-live` 通知
- [ ] 模态框打开时焦点管理正确
- [ ] 使用 `role` 属性标识组件角色

#### F. 错误处理 (Error Handling)

**必须检查：**
- [ ] 异步操作必须有 try-catch 或 .catch()
- [ ] 用户操作失败要有明确的错误提示
- [ ] 使用 Error Boundary 捕获组件错误
- [ ] 网络请求失败有重试或降级方案
- [ ] 避免空指针错误（使用可选链 `?.`）

**日志记录：**
- [ ] 生产环境不输出 `console.log`
- [ ] 错误日志包含足够的上下文信息
- [ ] 敏感信息不记录到日志

#### G. 样式规范 (Styling)

**Tailwind CSS：**
- [ ] 优先使用 Tailwind 原子类
- [ ] 复用样式通过 `@apply` 封装到 CSS 类
- [ ] 避免内联样式（除非动态计算）
- [ ] 响应式设计使用断点前缀（sm:, md:, lg:）
- [ ] 颜色使用主题变量（text-primary, bg-surface）

**CSS 最佳实践：**
- [ ] 避免使用 `!important`
- [ ] 避免过深的选择器嵌套（最多 3 层）
- [ ] 类名语义化，避免样式类名（如 `.red`）
- [ ] 动画使用 `transition` 而非 `animation`（除非必要）

#### H. 测试覆盖 (Testing)

**单元测试：**
- [ ] 工具函数有单元测试
- [ ] 复杂逻辑有单元测试
- [ ] 测试覆盖率 > 80%（关键路径）

**组件测试：**
- [ ] 关键组件有测试
- [ ] 测试用户交互（点击、输入）
- [ ] 测试边界情况（空数据、错误状态）

### 3. 生成审查报告

按以下格式输出审查结果：

```markdown
## Code Review 报告

### ✅ 通过检查 (X 项)
- 项目 1
- 项目 2

### ⚠️ 需要改进 (X 项)
- **[性能]** 问题描述
  - 位置：`文件路径:行号`
  - 建议：具体修复方案

### ❌ 必须修复 (X 项)
- **[安全]** 问题描述
  - 位置：`文件路径:行号`
  - 建议：具体修复方案

### 📊 总体评分
- 代码质量：⭐⭐⭐⭐⭐ (X/5)
- 性能优化：⭐⭐⭐⭐⭐ (X/5)
- 安全性：⭐⭐⭐⭐⭐ (X/5)
- 可维护性：⭐⭐⭐⭐⭐ (X/5)

### 💡 建议
- 总体建议 1
- 总体建议 2
```

### 4. 提供修复方案

对于发现的问题：
- 提供具体的代码修复示例
- 解释为什么这样修改
- 如果问题复杂，提供分步修复指南

## 审查优先级

**P0 - 必须修复（阻塞发布）：**
- 安全漏洞
- 功能性 bug
- 性能严重问题（页面卡顿、内存泄漏）

**P1 - 应该修复（影响质量）：**
- 代码规范违反
- 可访问性问题
- 错误处理缺失
- 测试覆盖不足

**P2 - 建议修复（优化改进）：**
- 代码可读性
- 性能优化建议
- 重构建议

## 注意事项

- 保持客观和建设性，避免主观批评
- 提供具体的修复建议，而非仅指出问题
- 优先关注影响用户的问题（功能、性能、安全）
- 尊重现有代码风格，除非违反最佳实践
- 如果不确定，说明原因并建议进一步讨论

## 工具使用规则

- 只使用 Read、Grep、Glob 工具读取代码
- 不使用 Edit、Write 工具修改代码（除非用户明确要求）
- 不使用 Bash 工具运行测试（除非用户明确要求）
- 不使用 Agent 工具（保持审查在主会话中）
